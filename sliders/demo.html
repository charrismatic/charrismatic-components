    Background:

    Frontend development is the area with the most frameworks, libraries, methedologies and personal opinions about the best or most correct way to do something. The end goal of all of these elements is to sucessfully and draw rectangles with special style properties in container that is viewed by an end user.

    While that might be an oversimplification, its not an exageration to say that this task can become incredibly complex in once things start connecting with other services and move around in ways that prevent you from knowing when you will get some information back from a thing.

    Kamran Ahmed, the maintainer of the “developer-roadmap[1]()” has written a pretty accurate write up of the current state of frontend develoment with some impressively detailed flowcharts on Medium. https://medium.com/tech-tajawal/modern-frontend-developer-in-2018-4c2072fa2b9c

    This is worth reading if you are about to start a development project and you are not a frontend developer (or you plan on hiring one).

    Some developers find a system that works for them and tend to stick with that method until they retire or that package disappears from github, which is understandable if you ever want to get anything done without a week of learning a totally new codebase.

    If you take a step back from the very large ecosystem of tools, you can see that there are just a few important stages of the frontend development lifecycle and the majority of tools out there are either interchangeable with each other, complimentary and can be used together, or do things at completely different stages of the project.

    So while there is a lot of debate and media coverage over these tools, as if they were competeing with each other, it really is not as important which tools you choose as much as, the members of your team use the same toolchains and the things you decide to add to a project actually work well together and not create a extra complexity and headaches.

    Among all of the things listed in the article above we have used all of them at some point and are comfortable working on a project that would require any combination of those tools.

    We also have a fairly strongly opinioanted set of preferences and well developed libraries for the things we thinkn are the best way to do things.

    Summary:

    Styles

    • SASS (.scss)

    LESS also comes up as an alternate choice quite often but the syntax and availability of frameworks to choose from almost always leads to refactoring LESS stylesheets to SCSS at some point in a project that is not SASS starting out. SASS is pretty well accepted industry standard, anything other these would proceed with caution to avoid roadblocks later in the project.

    Conventions

    Where possible we try to follow the principles of BEM, OOCS, and atomic style conventions, and accept the standard set of rules from style rules from Stylelint and htmlhint. Typically the standard default set is enough to provide some basic conventinons without creating a large complicated set of plugins and rules that require a bunch of maintence.

    Frameworks

    Where it makes sense css frameworks are added to projects, but never more than one framework should be used, and if possible only add the invdividual components directly, most modern frameworks offer the original source code now so you can pick and choose bits and pieces from serveral different libraries without bloating up your stylesheets or conflicts between them. Bootstrap4 and Material Design are the most often used style frameworks.

    Postprocessing

    The standard set of stylesheet transformations are always run through postcss before pushing to production. These are - csso (minify) - autoprefixer

    We have been experimenting with tree shaking using UNCSS to remove unused styles and tools like Plato to find overly complicated sections, but these tools only work very well for certain types of projects, for others they aren’t really able to help.

    Functionality

    For everything else there is a javascript package on npm or github…

    That’s a true statement but for almost everything that exists in the open source community, it could just as easily be built out using Vanilla Javascript and is easier to work into the existing codebase. We try to write plain ES6 js as much as possible and stay away from things like JQuery for frontend functionality.

    There are some very interesting tools available but once you become dependent on frameworks to manage your frameworks, things like Requirejs, webpack modules, etc. You can end up in very complicated but avoidable spots in the development cycle. Unless the project requires it or we’ve committed to using a particular type of application that will use it for 100% of the time we avoid most front javascript frameworks.

    MVCS

    With exception for MVC frameworks like React and Vuejs. These frameworks require a lot of extra tooling but allow you to do things that would not be possible any other way. Our current pick in this area has been Vue 3. React seems to have bigger adoption in the industry mainstream, but Vue has been growing and it looks better, and has some very unique tools avaiable that have been fun to work with.

    Build Tools / Tasks Runners

    The two most popular task frameworks are Gulp and Grunt. Between the two gulp seems to always have been more useful and cause fewer headaches when the project grew in complexity past the initial setup of the build tools.

    For the past year about updates in the NPM core have made it easy enough to remove these requirements altogether though. Using just the package.json run scripts and some very simple bash scripts to set up the environment you can get rid of almost every project build dependency. For things that are not that simple, a one or two file Nodejs script is usually enough to get a job done. This way of working has kept our build system very light and easy to use. Invdividual scripts are much easier to drop into a project than npm packages that bring along hundreds of other packages with them.

    Module Packagers

    When you need to assemble a module the current choice has been Rollup since it closely matches the javascript syntax for native modules once they become fully adopted in the future. Also we have had less experience with Parcel but it seems to be worth using.

    Anything not covered here will roll over into the backend dev discussion.
